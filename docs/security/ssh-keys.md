# SSH keys

How the keystone fleet manages SSH private keys for human users (`ncrmro` etc.) and agent users (`agent-luce` etc.) so that `git push`, `ssh`, and commit signing work with zero prompts after first sign-in on each host.

This document covers user/agent identities. It does **not** cover root SSH (handled by `keystone.hardwareKey.rootKeys` and a YubiKey-resident key) or host keys (managed by OpenSSH).

## Goal

Per host, per user/agent identity:

- One sign-in (login for humans, boot for agents).
- The user's SSH private key is loaded into a running `ssh-agent` and stays there for the session.
- Every subsequent SSH or git operation in that session uses the agent — no passphrase prompts, no manual `ssh-add`.

## Threat model

In scope:

- Backup leaks (rsync to a less-trusted target, exported home archive).
- Theft of the `agenix-secrets` repo without a corresponding age recipient.
- Lost or stolen powered-off machine.
- Naive `find $HOME -name 'id_*'` style snooping.

Out of scope:

- Same-UID malicious code. Anything running as your user can read both `~/.ssh/id_ed25519` and `/run/agenix/${hostname}-${username}-ssh-passphrase`. The realistic isolation boundary is the user account, not the process.
- Root compromise on a recipient host. Once root, all decrypted secrets in `/run/agenix/` are readable.

Honest limit: only hardware-backed keys (YubiKey FIDO2, `id_ed25519_sk_*`) eliminate readable on-disk private-key material. Software keys always leave _something_ readable to the user. `sshAutoLoad` makes the threat model explicit and consistent across the fleet, but it doesn't change that.

## Three knobs

SSH key UX on a host is the composition of three independent settings. Confusing one for another is the source of most "why does it keep prompting?" questions.

| Knob | Setting | What it does | Lifetime |
|------|---------|--------------|----------|
| 1. Agent process runs | `services.ssh-agent.enable` (home-manager) | Starts `ssh-agent.service` at user-manager start. Sets `SSH_AUTH_SOCK`. | User session |
| 2. Key auto-loaded | `keystone.os.users.<name>.sshAutoLoad.enable` | Oneshot at login that `ssh-add`s the user's key using the agenix passphrase. | Until logout |
| 3. Implicit-add TTL | `programs.ssh.matchBlocks."*".addKeysToAgent` | When SSH itself loads a key during a connection (you typed the passphrase at the terminal), how long the agent keeps it. | TTL value |

How they compose:

| Agent (1) | sshAutoLoad (2) | TTL (3) | UX |
|-----------|-----------------|---------|----|
| off       | off             | n/a     | Every operation prompts. `ssh -A` has no destination — agent forwarding to this host doesn't work. |
| on        | off             | unset/`"yes"` | First passphrase prompt per session, then cached forever. |
| on        | off             | `"1h"`  | Re-prompt every hour of active use. |
| on        | on              | any     | Zero prompts. The TTL is irrelevant: `ssh-add` adds with no expiry, ignoring `AddKeysToAgent`. |

For a workflow where agents do most pushes and the human rarely triggers SSH interactively (the common case for ocean/maia/mercury), `(1)=on, (2)=on` is the only configuration that doesn't degrade to "type the passphrase at every session start."

## Where the private key lives

| Identity | Path | Owner / mode |
|----------|------|--------------|
| Human user | `~/.ssh/id_ed25519` (passphrase-encrypted) | user / 0600 |
| Agent user | `/run/agenix/${username}-ssh-key` (decrypted from agenix at boot) | user / 0400 |

Human users keep their key at the canonical `~/.ssh/` path. Every tool that needs an SSH key (`git`, `ssh`, IDEs, rsync over ssh) looks here by default — moving it elsewhere means overriding `IdentityFile` everywhere. The key is passphrase-encrypted at rest, so a backup leak or stolen disk doesn't yield a usable key without the matching passphrase.

Agents have no human-managed `~/.ssh/` workflow: their keys are generated by `agentctl provision` and stored in `agenix-secrets` from the start, decrypted to `/run/agenix/` at boot.

### Why not move the human key into agenix too

Tempting, but doesn't actually buy more security:

- Same-UID processes can still read `/run/agenix/${hostname}-${username}-ssh-key` on the running host. The realistic isolation boundary doesn't move.
- The "key path" change is security-through-obscurity. An attacker who knows about agenix knows where to look.
- Defense-in-depth is already correct: stolen encrypted key file is useless without the passphrase; stolen passphrase is useless without the key file. Compromise requires both, and the passphrase is only decrypted on hosts whose age recipient is in `secrets.nix`.
- Breaks tooling defaults (`git`, `ssh`, GUI clients, scripts) for no real gain.

If you need true "no-readable-private-key" guarantees, use a hardware key. See [hardware-key escape hatch](#hardware-key-escape-hatch).

## Where the passphrase lives

Agenix secret: `${hostname}-${username}-ssh-passphrase.age`.

At runtime, decrypted to `/run/agenix/${hostname}-${username}-ssh-passphrase` (tmpfs, mode 0400, owner=the user, wiped on reboot).

### Naming rationale

| Component | Why |
|-----------|-----|
| `${hostname}` segment | The same human (`ncrmro`) has different keys on `ocean`, `ncrmro-laptop`, `ncrmro-workstation`. Disambiguates per host. |
| `${username}` segment | Multiple users may exist on one host (e.g. `ncrmro` + `agent-luce` both on ocean). Disambiguates per identity. |

Agents adopt the same convention (`ocean-agent-luce-ssh-passphrase` etc.) even though agent usernames are already host-bound by construction — consistency across the fleet beats saving the redundant segment.

## Why agenix, not gnome-keyring / pam_ssh / KWallet

Desktop keyrings (gnome-keyring with PAM unlock, KWallet, the now-defunct `pam_ssh`) deliver the same UX in principle: log in, your SSH key is in the agent, no further prompts. Keystone doesn't use them.

- **Threat model.** Desktop-keyring auto-unlock derives the authority from your **login password**. A weak password, a keylogger, or password reuse compromises every SSH key it holds. Agenix binds authority to the host's age recipient — usually the host's SSH key plus admin recipients (YubiKey-resident on `keystone.hardwareKey.rootKeys` hosts). A recovered login password buys nothing.
- **Fleet shape.** gnome-keyring needs a session manager and dbus. Pulling that stack onto headless servers (`ocean`, `maia`, `mercury`) just to unlock a key is the wrong direction.
- **Agents.** Non-interactive system users (`agent-luce`, `agent-drago`) don't go through interactive PAM at boot. They'd need a separate mechanism — at which point the fleet has two unrelated SSH-unlock systems.
- **Maintenance.** `pam_ssh` is effectively unmaintained. gnome-keyring's own SSH integration was deprecated around 2018–2020; modern GNOME defers to upstream `ssh-agent`. The PAM-unlock part (the keyring as a generic password store) still works fine — it's the SSH-specific path that rotted.

If you run a desktop, gnome-keyring is still useful for browser secrets, wifi credentials, etc. Let it own those; let agenix + `sshAutoLoad` own SSH specifically.

## Two service shapes

Humans and agents use the same secret-naming scheme but different systemd shapes.

| Identity | Service | Type | Triggered by | Why |
|----------|---------|------|--------------|-----|
| Human | `systemd.user.services.ssh-auto-load` | oneshot | `WantedBy = default.target` | Fires when the user systemd manager starts. For non-linger humans that's first interactive login per boot — the closest systemd gets to "on sign-in." |
| Agent | `systemd.services.agent-${name}-ssh-agent` | simple (daemon) | `WantedBy = multi-user.target`, `Restart = always` | Agent processes need SSH ready before any human logs in. Always-on is intentional. |

References: `keystone/modules/terminal/ssh-auto-load.nix:81` (human), `keystone/modules/os/agents/ssh.nix:147` (agent).

Why not collapse to one shape: humans without `linger` only want their agent during interactive sessions; agents need it 24/7. A single shape forces one side to compromise.

## Enrollment — humans

You already have `~/.ssh/id_ed25519` (passphrase-protected). No regeneration needed; the enrollment is "tell agenix the passphrase you already have."

```bash
# 1. Enroll the passphrase into agenix.
cd agenix-secrets
agenix -e secrets/${hostname}-${username}-ssh-passphrase.age
# Paste the passphrase, save, exit your editor.

# 2. Authorize the host (and admins) to decrypt it.
# Edit secrets.nix:
#   "secrets/${hostname}-${username}-ssh-passphrase.age".publicKeys =
#     adminKeys ++ [ systems.${hostname} ];

# 3. Commit and push the agenix-secrets change.
git add secrets.nix secrets/${hostname}-${username}-ssh-passphrase.age
git commit -m "Enroll ${hostname} ${username} SSH passphrase"
git push

# 4. From ks-config, update the lock.
cd ../ks-config
nix flake update agenix-secrets
git add flake.lock
git commit -m "chore(deps): bump agenix-secrets — ${hostname} ${username} ssh passphrase"
```

Then in the host's NixOS config:

```nix
keystone.os.users.${username}.sshAutoLoad.enable = true;
```

Rebuild and re-login:

```bash
ks update ${hostname}
# Log out and back in (or reboot) so the user manager starts ssh-auto-load.service.
```

Verify (see [verification](#verification-on-a-host)). Zero prompts on this host from now on.

## Enrollment — agents

Use the provisioning helper:

```bash
agentctl provision <agent-name>
```

`agentctl` (see `keystone/modules/os/agents/scripts/agentctl.sh:861+`) generates a fresh ed25519 keypair, generates a passphrase, encrypts both as agenix secrets under `${host}-${username}-ssh-{key,passphrase}.age`, updates `secrets.nix`, and prints the `age.secrets.*` snippets to add to the host's NixOS config.

## Lifetime nuances

- **`ssh-add` (what `sshAutoLoad` invokes) has no expiry.** The key stays in the agent until the agent dies — i.e., until logout (humans) or reboot/service stop (agents).
- **`AddKeysToAgent` is only for keys SSH adds implicitly.** When you run `git push` and SSH itself reads `~/.ssh/id_ed25519`, prompts for the passphrase, and tells the agent to remember it, the `AddKeysToAgent` TTL applies to that key. `ssh-add` ignores the setting entirely.
- **`AddKeysToAgent = "1h"`** (the current desktop default in `keystone/modules/desktop/home/components/ssh-agent.nix:19`) only matters when `sshAutoLoad` is off. With `sshAutoLoad` on, the key was added via explicit `ssh-add` and has no TTL.

If you keep needing to `ssh-add` at the start of every session, the cause is almost certainly "no `sshAutoLoad` on this host." The `1h` TTL is downstream of that — if SSH never gets a chance to implicitly add your key (because you don't run interactive SSH between agent-driven activity), the cache window doesn't help.

## Hardware-key escape hatch

For any account that can use a hardware key, the entire `sshAutoLoad` mechanism becomes unnecessary:

- Root SSH across the fleet uses YubiKey-resident keys, wired via `keystone.hardwareKey.rootKeys` (see `modules/keystone/os.nix:56`).
- For human users, `id_ed25519_sk_yubi-*` keys (e.g. on ocean: `~/.ssh/id_ed25519_sk_yubi-black`) require a physical touch and never leave the YubiKey. No passphrase, no agenix, no `sshAutoLoad` configuration.

`sshAutoLoad` exists only for accounts that can't (or don't yet) use hardware keys: agents, and humans on hosts where touching a YubiKey for every git operation is too disruptive.

## Verification on a host

Humans:

```bash
systemctl --user status ssh-agent.service ssh-auto-load.service
echo $SSH_AUTH_SOCK
# Expected: /run/user/$(id -u)/ssh-agent (or similar non-empty path)

ssh-add -l
# Expected: one line per loaded identity (your ed25519 key)
```

Agents (run as root or via `agentctl`):

```bash
systemctl status agent-${name}-ssh-agent.service
sudo -u agent-${name} env SSH_AUTH_SOCK=/run/agent-${name}-ssh-agent/agent.sock ssh-add -l
```

End-to-end:

```bash
# Should succeed without prompting.
git -C ~/some-forgejo-clone push --dry-run
```

## Troubleshooting

### `bin/ks-dev` fails with "Permission denied (publickey)" while fetching `vega.git`

This happens on hosts without `sshAutoLoad` enabled. `nixos-rebuild` runs the flake evaluation under `sudo`, and root has no SSH agent of its own to fetch private flake inputs. Commit `634e917`'s `--preserve-env=SSH_AUTH_SOCK` works only if **you** have an agent for sudo to forward.

Immediate unblock (no rebuild required): SSH into the host with agent forwarding.

```bash
# From your laptop with the SSH key loaded:
ssh -A ${hostname}
# Then run ks-dev. sudo forwards SSH_AUTH_SOCK to root, root fetches via your agent.
```

Permanent fix: complete the [human enrollment flow](#enrollment--humans) so the host has its own agent + key at login. `ssh -A` is no longer needed afterward.

### `ssh-add -l` says "Could not open a connection to your authentication agent"

The ssh-agent service isn't running. Either:

- `services.ssh-agent.enable = true` was never set (knob 1). On a terminal-only host without `sshAutoLoad`, this is the default — nothing turns the agent on. Enable `sshAutoLoad` or set `services.ssh-agent.enable` directly in home-manager.
- The user manager wasn't started (you SSHed in non-interactively or `linger` is off and there's no live session). Check `loginctl show-user ${username}`.

### Re-prompt every hour despite enrollment

Almost always: `sshAutoLoad` isn't actually enabled for this user on this host. Confirm with `systemctl --user status ssh-auto-load.service`. If it's "inactive (dead)" or not found, the option isn't wired through to home-manager — check that `keystone.os.users.${username}.sshAutoLoad.enable = true` is set on the host config and that you've rebuilt.

If `ssh-auto-load.service` shows `Result: exit-code` and a stale-passphrase warning, the agenix secret was enrolled with the wrong passphrase. Re-enroll (`agenix -e ...`) with the actual passphrase used to encrypt `~/.ssh/id_ed25519`.
